## Featured: `Tool(param:value)` Permission Syntax

Shipped in v2.1.178, this new permission rule syntax lets you match against a tool's input parameters — not just the tool name. Wildcards (`*`) are supported.

**Why it matters:** Until now, permission rules could only allow/deny entire tools. If you wanted to let Claude spawn Haiku subagents but block Opus ones, you couldn't. Now you can write granular rules that match specific parameter values.

**Usage examples in `settings.json`:**

```json
{
  "permissions": {
    "deny": [
      "Agent(model:opus)",
      "Bash(command:rm *)"
    ],
    "allow": [
      "Edit(file_path:src/**)",
      "Agent(model:haiku)"
    ]
  }
}
```

- `Agent(model:opus)` — block Opus subagents while allowing cheaper models
- `Bash(command:rm *)` — deny any bash command starting with `rm`
- `Edit(file_path:src/**)` — auto-allow edits only within `src/`

This pairs well with auto mode: you can let the classifier handle most decisions while hard-blocking specific high-risk parameter combinations.
