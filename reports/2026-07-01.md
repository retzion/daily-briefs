## Featured: Cache Diagnostics (`diagnostics.previous_message_id`)
A new public beta on the Claude API that tells you *why* your prompt cache missed, instead of leaving you to guess.

**The problem:** Prompt caching can cut cost and latency dramatically, but a reordered tool, an interpolated timestamp in your system prompt, or a re-serialized assistant turn silently invalidates the cache. The only signal was `cache_read_input_tokens` dropping to zero — no explanation why.

**How it works:** Pass the `id` from your previous API response as `diagnostics.previous_message_id` on the next request. The API compares lightweight fingerprints (hashes only — no raw content stored) and returns a `cache_miss_reason` with the type of divergence:

| Reason | Meaning |
|---|---|
| `system_changed` | Your system prompt changed (e.g., interpolated timestamp) |
| `tools_changed` | Tools added/removed/reordered, or schemas serialized differently |
| `messages_changed` | Earlier message was edited/truncated instead of appended |
| `model_changed` | Different model selected (cache is per-model) |

**Quick start:**
```python
# Add beta header and pass previous response ID
r2 = client.beta.messages.create(
    model="claude-sonnet-5",
    messages=[...],
    diagnostics={"previous_message_id": r1.id},
    betas=["cache-diagnosis-2026-04-07"],
)
if r2.diagnostics and r2.diagnostics.cache_miss_reason:
    print(f"Cache miss: {r2.diagnostics.cache_miss_reason.type}")
```

Especially valuable for agentic systems where multi-turn conversations with tools make cache invalidation easy to introduce accidentally.
